# Reviewer Agent 检查清单

> **适用角色**：Reviewer Agent / 质量复核官
> **执行逻辑**：凡是标注为 `[必检]` 的项未通过，必须拦截并打回 Auditor Agent 重审。

---

## 一、审计完整性评分 (Completeness)

- [ ] `[必检]` **关键要素提取**：报告是否准确提取了以下核心信息？
  - 合同主体（甲方、乙方）
  - 合同金额
  - 生效日期
  - 租赁/服务期限

- [ ] `[必检]` **盲区扫描**：对于长达 5 页以上的合同，审计项是否覆盖了全文？
  - 判定标准：若审计点少于 5 处，则视为漏检风险

- [ ] **违约链条检查**：是否同时覆盖了以下三种核心违约场景？
  - 逾期付款
  - 逾期交付
  - 提前解约

---

## 二、风险逻辑严密性 (Logic & Depth)

- [ ] `[必检]` **法律红线对标**：涉及违约金比例时，是否对照了 `legal-guidelines.md` 中的法定上限？
  - 检查点：是否提示了超过 30% 实际损失的风险

- [ ] **逻辑矛盾识别**：审计结论中是否存在自相矛盾？
  - 示例：一边说合同合规，一边又列出致命缺失项

- [ ] **管辖地合理性**：是否检查了争议解决地与以下地点之间的距离成本风险？
  - 合同履行地
  - 当事人所在地

---

## 三、输出质量与稳定性 (Deliverable Quality)

- [ ] `[必检]` **格式对齐**：输出是否严格符合 `assets/report-template.md` 的 Markdown 表格结构？
  - 检查：是否有乱码或未闭合的标签

- [ ] **证据溯源**：每一项"风险描述"是否都指明了其在原合同中的具体条款序号？
  - 示例格式：见合同第五条第 2 款

- [ ] **去幻觉校验**：报告中提到的所有"风险条款"，在原合同原文中是否真实存在？
  - ⚠️ 严禁 Agent 凭空编造合同中没有的条款进行审计

---

## 四、改进建议的建设性 (Actionability)

- [ ] **意见闭环**：每一条风险点是否都配有具体的"修改建议"或"补充条款示例"？

- [ ] **语气专业度**：修改建议是否使用了法务专业用语？
  - ✅ 正确示例："连带责任"、"违约责任"
  - ❌ 错误示例：口语化表达

---

## 🚫 一票否决项 (Veto Criteria)

> 如果出现以下任何一种情况，质检员必须给出 **0 分** 并 **强制打回**：

| 否决项 | 描述 |
|--------|------|
| **胡言乱语** | 输出中包含 AI 自我介绍、多余的解释或非 Markdown 格式内容 |
| **致命漏检** | 合同中明显存在"哈尔滨仲裁（针对深圳公司）"等管辖逻辑坑，但审计报告未标记 |
| **JSON/格式崩溃** | 下游系统需要解析结果，但输出的结构损坏 |

## 💡 Reviewer Workflow

将以下编排逻辑添加到 Reviewer Agent 的 `SKILL.md` 中：

```markdown
## Reviewer Workflow:

1. 读取 Auditor Agent 生成的 `Draft_Report.md`
2. 加载 `@references/reviewer-checklist.md`
3. 对照清单逐项打分
4. 判定逻辑：
   - IF 存在 `[必检]` 项未通过：
     - 输出具体的"打回理由"并要求 Auditor 重新生成
   - ELSE:
     - 输出最终的 `Final_Audit_Report.md`
```
